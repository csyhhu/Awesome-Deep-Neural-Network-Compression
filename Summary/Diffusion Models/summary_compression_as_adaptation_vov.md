> 本文是一个**视觉/视频压缩（编解码）方法**，与 HEVC/H.266 等传统编解码器同类。输入是一段**已有的原始视频/图像**，输出是压缩后的比特流，解码端从比特流重建回原始内容。

## 1. 核心思想

本文提出了一种全新的视觉表示框架：**将视觉信号编码为一个函数（function），而非传统的像素、潜变量或 token**。该函数通过在大规模扩散生成模型上附加**低秩适应（LoRA）参数**来参数化，从而形成一种**隐式视觉表示（Implicit Visual Representation）**。

核心动机：
- 现代大规模视觉生成模型（如 Diffusion / Flow Matching 模型）已经通过大规模训练学到了丰富的视觉知识
- 但现有的视觉表示（像素、潜变量、token）都在模型外部，无法直接利用这些知识
- 本文提出直接"压缩生成过程本身"，而非压缩信号本身

## 2. 方法详解

### 2.1 隐式视觉表示学习

**整体流程**：
1. 给定视觉信号 \(x\)（图像或视频），先用 VLM（如 GPT-5.1）获取详细描述 \(c\)
2. 在冻结的大规模视觉生成模型上，以 \(c\) 为条件，学习 LoRA 适应参数
3. 训练目标为 flow-matching 损失：

\[
\mathcal{L}_{\text{FM}}(\theta) = \mathbb{E}_{t,\epsilon}\left[\|v_\theta(x_t, t) - (\epsilon - x)\|^2\right]
\]

其中 \(x_t = (1-t)x + t\epsilon\)，\(t \sim \text{Unif}[0,1]\)，\(\epsilon \sim \mathcal{N}(0,I)\)

**理论解释（MDL 视角）**：
从最小描述长度角度，该训练目标等价于在路径空间中寻找与预训练模型生成的路径测度 \(\mathbb{P}\) 差异最小的新测度 \(\mathbb{P}'\)：
\[
\min_{\mathbb{P}'} D_{\text{KL}}[\mathbb{P}'\|\mathbb{P}], \quad \text{s.t. } x_0 = x
\]
最优解是预训练扩散过程的条件过程（Doob's h-transform），flow-matching 目标的最优解正好恢复了这个解。

### 2.2 单向量压缩（One-Vector Adaptation）

为解决实际压缩中的码率问题，提出两个关键技术：

**① LoRA + 哈希映射（Hashing）**：
- 使用 LoRA 对模型中所有 Transformer 层的 Q/K/V/输出投影层进行低秩适应
- 将所有 LoRA 参数通过伪随机数生成器（PRNG）映射到单一共享向量 \(\mathbf{v} \in \mathbb{R}^{1\times k}\) 中
- 类似 Uni-LoRA 的思想，实现参数共享

**② 熵约束量化**：
- 对单向量 \(\mathbf{v}\) 引入可学习的 scale 参数并做均匀量化
- 训练时使用加性均匀噪声实现可微分码率估计
- 采用因子化熵模型（factorized entropy model）建模向量元素的分布
- 最终每个参数约 1-3 bits，熵模型额外开销可忽略（<1%）

该压缩框架称为 **VOV（Vision/Video in One Vector）**。

### 2.3 推理时缩放（Inference-Time Scaling）

函数式表示的核心优势：编码后函数仍可控，可以在不改变压缩表示的情况下进一步提升质量。

**核心算法——基于重要性采样的 Sequential Monte Carlo**：

1. 编码器和解码器共享压缩后的适应参数和 PRNG
2. 使用 SDE 形式进行去噪：
   \[
   p(x_{t_{n-1}}|x_{t_n}) = \mathcal{N}(x_{t_{n-1}}|\mu_\theta(x_{t_n}), \frac{2t_n}{1-t_n}\Delta t)
   \]
3. 编码器在每个时间步从 \(p(x_{t_{n-1}}|x_{t_n})\) 采样 \(M\) 个候选粒子
4. 以最优去噪核 \(p^*(x_{t_{n-1}}|x_{t_n})\)（通过训练目标解析形式获得）作为目标分布，进行重要性采样选择最优粒子
5. 将选中粒子的索引编码为少量边信息传输给解码器
6. 解码器利用共享 PRNG 确定性复现编码器选择

该过程可同时从两个互补视角理解：
- **推理时精炼**：用重要性采样精炼学到的生成动力学
- **Diff-C 视角**：将带有适应的扩散模型作为更强的先验，降低相对熵编码复杂度

### 2.4 压缩与生成的统一

LoRA 适应参数直接附着在生成模型上，可作为"视觉记忆"：
- 修改文本提示即可实现个性化编辑（颜色、数字变化等）
- 支持图像合并与分辨率变化
- 同时支持忠实重建和灵活编辑

## 3. 实验结果

### 3.1 表示能力
- 即使 rank=1 的 LoRA 也能实现合理重建
- 单向量表示中，增大向量维度 \(k\) 持续提升重建质量
- 固定 \(k\) 时，增大 LoRA rank 反而可能降低质量（高秩引入更密集纠缠的参数更新，在固定哈希方案下难以保留）

### 3.2 视频压缩（VOV）
- **基模型**: Wan-2.1 (1.3B)，分辨率 832×480，81 帧
- **数据集**: UVG, HEVC B/C/E
- **对比方法**: H.265/HM, H.266/VTM, DCVC-RT, GLC-Video 等
- **结果**：
  - 在极低码率下，DISTS 和 FVD 指标显著优于现有方法
  - 视觉质量明显更好，尤其在细节和纹理保留方面
  - 利用扩散模型的时序先验，视频具有更好的时间一致性

### 3.3 推理时缩放效果
- 无分支时增加去噪步数几乎没有提升
- 每步引入多样本可大幅提升质量，样本越多效果越好
- 增加步数在有多样本时变得有效（有时比增加样本数更有效）
- 两种缩放策略：100 步 + \(2^{18}\) 样本/步（仅编码端缩放），或 1000 步 + \(2^{10}\) 样本/步

### 3.4 纯缩放压缩
即使不学习适应参数，直接在预训练模型上做纯推理时缩放也能取得强压缩性能，但计算开销更大，且无法支持编辑和连续控制。

### 3.5 失真-感知权衡
通过提前终止去噪过程并利用 Tweedie 公式重建，可实现免费的推理时失真-感知（DP）权衡。

## 4. 关键贡献

1. **新范式**：首次提出将视觉信号压缩为扩散生成模型的适应函数（隐式表示）
2. **单向量压缩**：通过哈希和熵约束将整个 LoRA 适应压缩为单个紧凑向量
3. **推理时缩放**：利用函数式表示的特性，通过重要性采样在编码端精炼重建质量
4. **统一框架**：桥接压缩与生成，适应参数可用作个性化生成的"视觉记忆"

---

## 5. 局限性与未来方向

**局限性**：
1. 受限于扩散基础模型的能力，偶尔出现语义不匹配（尤其是视频中的文字）
2. 编码过程（过拟合隐式表示）时间较长，类似于 INR 方法
3. 适应参数间相关性通过随机哈希映射捕获，效率有限

**未来方向**：
- 学习**摊销编码器**（amortized encoder）和**摊销解码器**（从向量到完整 LoRA），加速编码
- 更好地捕获适应参数间的相关性，进一步提升压缩率

---

## 6. 与现有工作的关系

- **INR 压缩**（COIN, NeRV, NVRC 等）：共享"函数式表示"思想，但本文利用大规模预训练模型的知识，而非从零训练小网络，在感知质量上显著优于 NVRC
- **LoRA 个性化生成**（DreamBooth, Custom Diffusion 等）：同样使用 LoRA 适应，但本文将其重新定义为压缩表示并扩展到视频
- **推理时缩放**（Diffusion SMC 等）：将 SMC 思想适配到压缩场景，并创新性地在编码端使用
- **相对熵编码**（Diff-C, REC）：本方法可视为 Diff-C 的变体，以适应后的扩散模型为更强先验

## 7. 问答讨论

### Q: 用一个例子说明训练和推理的过程

以压缩一段 **81帧、480p 视频**（内容：一只橘猫在木地板上玩红色毛线球）为例，完整流程如下：

#### 一、训练/编码阶段

**Step 1 — 生成语义锚点（Captioning）**

用 VLM（如 GPT-5.1）生成详细描述，作为扩散模型固定的条件输入：
> *"An orange tabby cat batting a red ball of yarn on a wooden floor, with sunlight streaming through a window."*

**Step 2 — LoRA 适应训练（把视频"写入"模型）**

在冻结的 Wan-2.1 (1.3B) 上，对所有 Transformer 层的 Q/K/V/输出投影层插入 LoRA（rank=4）。每次训练迭代：

- 随机采样时间 `t ∈ [0,1]`，噪声 `ε ~ N(0,I)`
- 构造带噪视频 `x_t = (1-t)·原视频 + t·ε`
- 将 `(x_t, t, caption)` 送入「冻结模型 + LoRA」
- 模型输出速度场 `v_θ(x_t,t)`，目标为 `(ε - 原视频)`
- 计算 MSE 损失，仅反向传播更新 LoRA 参数

约 1000 步后，LoRA 学会了从噪声生成该视频的去噪路径。**此时 LoRA 参数本身就是视频的隐式表示**。

**Step 3 — 哈希压缩为单一向量**

```
LoRA_A₁,B₁, A₂,B₂, ... →（PRNG 哈希映射）→ v ∈ R^{1×k}（如 k=65536）
```

**81 帧视频 = 一个向量**。参数量从百万级降至万级。

**Step 4 — 量化与熵编码**

```
v (float32) → 均匀量化 → 因子化熵模型编码 → 比特流
```

最终码流 = 向量比特流 + caption 文本 + 熵模型参数（后两项开销 < 1%）。

#### 二、推理/解码阶段

**基础解码（无 Scaling）**：

```
比特流 → 熵解码 → v̂ → PRNG 展开 → 完整 LoRA → 从纯噪声走 N 步 ODE/SDE → 重建视频
```

**推理时缩放解码**（核心亮点：编码端多干活，解码端轻量复现）：

编码端（一次性完成，产出边信息）：

```
从 x₁~N(0,I) 出发，逐时间步 tₙ → tₙ₋₁:

每个时间步：
  ① 共享 PRNG 从 p(x_{tₙ₋₁}|x_{tₙ}) 采样 M 个候选粒子
  ② 计算重要性权重 w^(m) ∝ p*(x_{tₙ₋₁}^(m)|x_{tₙ}) / p(x_{tₙ₋₁}^(m)|x_{tₙ})
     其中 p* 是最优去噪核（编码端有原始视频 x，可解析计算）
  ③ 按权重采样选最优粒子，编码其索引（log₂M bits/步）

输出：N × log₂M bits 的边信息（索引序列）
```

解码端（快速）：

```
用相同 PRNG 种子，每步生成相同 M 个候选
  → 直接按收到的索引选取（无需计算 p*！）
  → 继续下一时间步
```

解码端不需要原始视频，不需要计算重要性权重，只需按索引机械选取——这就是解码端轻量的原因。

#### 直观对比

| 环节 | 传统编解码 | VOV（本文） |
|------|-----------|------------|
| 编码对象 | 像素/潜变量的压缩码 | 生成过程的函数参数（LoRA → 单向量） |
| 解码方式 | 固定解码器重建 | 适应后的扩散模型从噪声生成 |
| 额外控制 | 无 | 推理时缩放、提前停止、prompt 编辑 |
| 编码代价 | 低 | 高（需过拟合 + 可选 Scaling） |
| 解码代价 | 低 | 中等（扩散采样 N 步） |

### Q: LoRA 中的 A 和 B 是什么？

LoRA 将权重更新矩阵 ΔW 分解为两个低秩因子矩阵的乘积：

```
ΔW = A · B
```

其中：
- **A ∈ R^{m×r}**：下投影矩阵，将输入维度从 n 压缩到秩 r
- **B ∈ R^{r×n}**：上投影矩阵，将 r 维恢复回原始输出维度 m
- **r ≪ min(m,n)**：秩，远小于原始维度

适应后的权重：**W = W₀ + A·B**（W₀ 冻结不变，只训练 A 和 B）

**具体数值例子**：W₀ 为 4096×4096（约 16M 参数），取 r=4：

| 组件 | 形状 | 参数量 |
|------|------|--------|
| 原始 W₀ | 4096 × 4096 | 16,777,216 |
| A | 4096 × 4 | 16,384 |
| B | 4 × 4096 | 16,384 |
| **LoRA 合计** | — | **32,768** |

压缩比约 512 倍。A 通常用随机高斯初始化，B 初始化为零矩阵（保证训练从恒等映射开始）。

在本文中，Wan-2.1 每层 Transformer 的 **Q、K、V、输出投影** 各有一对 (A,B)，全部层的所有 (Aᵢ,Bᵢ) 再通过 PRNG 哈希映射到一个共享向量 v 以进一步压缩。

### Q: Step 4 的量化与熵编码在做什么？LoRA 哈希后的量化想学到什么？

**问题**：Step 3 哈希得到的向量 v（如 65536 维 float32，约 256 KB）仍然有冗余——每个元素用 32 位浮点表达过于"奢侈"，模型实际不需要这么高的精度就能重建。

**目标**：将每元素从 32-bit float 压缩到 1-3 bits。

**流程**：

```
v (float32, k=65536)
    ↓ ① 用可学习 scale s 归一化到合适范围
    ↓ ② 均匀量化：浮点 → 小整数（如 8 级 = 3 bits/元素）
    ↓ ③ 熵编码：按各值出现概率分配变长编码（算术编码）
    ↓
比特流（约 100K bits ≈ 12 KB）
```

**三个学习目标**：

| 学习对象 | 参数量 | 作用 |
|---------|--------|------|
| **v 的量化值** | k 维 × 1-3 bits | 视频内容的主要载体 |
| **scale s** | 1 个标量 | 控制量化动态范围：太大会浪费精度，太小会截断信息 |
| **熵模型参数 ψ** | 8 个标量 | 建模元素值的概率分布 p(vᵢ\|ψ)，让熵编码器给高频值分配短码 |

**训练技巧——加性噪声替代量化**：`round()` 梯度为零，训练时用加均匀噪声 `U[-0.5, 0.5]` 替代，分布近似一致且可微分（Ballé et al. 2017 标准做法）。

**损失函数**：

```
Loss = 重建误差（Flow-Matching loss） + λ × Rate
Rate = -Σᵢ log₂ p(vᵢ | ψ)  （熵模型估算的码率）
```

**压缩效果**（81 帧 480p 视频）：

| 阶段 | 数据量 |
|------|--------|
| 原始视频 | 81×832×480×3×8 ≈ 777M bits |
| 完整 LoRA 参数 | 数百万 float32 ≈ 几十 M bits |
| 哈希向量 v (float32) | 65536×32 ≈ 2.1M bits |
| **量化+熵编码后** | **≈ 100K bits** |
| caption + 熵模型参数 | < 1% 额外开销 |

**核心思想**：量化本质是在"去精度"与"保质量"之间找平衡——v 的每个维度不需要表达精确浮点值，模型学到的是粗粒度值 + 扩散模型先验就足以重建高质量视频。

### Q: 最终这个 Prompt 会产生一个一维的表达？维度是多少？

是的，最终压缩结果是一个一维向量 **v ∈ R^{1×k}**。k 的具体值取决于场景和模型：

| 场景 | 模型 | k 值 |
|------|------|------|
| 图像表示实验 | Qwen-Image-20B | 2¹⁵ / 2¹⁶ / 2¹⁷ / 2¹⁸（32K ~ 262K） |
| 视频表示实验 | Wan-2.1-1.3B | 2¹⁵ / 2¹⁶ / 2¹⁷ / 2¹⁸（32K ~ 262K） |
| **视频压缩 VOV（主实验）** | **Wan-2.1-1.3B** | **2¹⁷ = 131,072** |
| 视频编辑 | Wan-2.1-14B | 2¹⁹ = 524,288 |

以主实验为例：**81帧 480p 视频 → 一个 131,072 维向量**，量化后约 1-3 bits/维，总共约 16-48 KB。对比原始视频（~777M bits），压缩近 **20,000 倍**。

k 越大重建质量越高但码率也越高，论文通过不同 λ（码率权重）控制码率曲线的工作点。更大的基础模型（14B）需要更大的 k（524K）来充分表达其 LoRA 适应。

### Q: 推理时也会得到完整的 LoRA 参数吗？

是的。编码/解码完全对称：

```
编码端：原始视频 → LoRA 过拟合训练 → 完整 LoRA → 哈希压缩为 v → 量化+熵编码 → 比特流
解码端：比特流 → 熵解码 → v̂ → 相同 PRNG 哈希展开 → 完整 LoRA → 注入冻结模型 → 扩散采样 → 重建视频
```

关键在于 PRNG 哈希映射是**确定性的、可逆的展开**——给定同一个 v̂ 和同一个随机种子，就能精确恢复出所有层的全部 Aᵢ/Bᵢ 矩阵。编码端「学」出 LoRA（通过过拟合训练），解码端「重建」出 LoRA（通过哈希展开），**解码端不再训练**，只做熵解码 + 确定性展开 + 扩散采样。

### Q: 解码端（推理时）的比特流从哪里来？推理时也需要编码吗？

**推理时不需要编码。** 比特流是编码端提前生成好的，解码端只负责消费。

这是标准的编解码流水线，不是同一台机器上自产自销：

```
编码端（离线，有原始视频）：
  原始视频 → LoRA 训练 → 量化+熵编码 → 比特流（产物）

解码端（推理时，无原始视频）：
  比特流（读入） → 熵解码 → 展开 LoRA → 扩散采样 → 重建视频
```

类比播放 `.mp4` 文件：编码端（服务器）提前把视频压成 `.mp4`，解码端（播放器）读入并解码播放——播放器不需要原始视频，也不需要重新编码。比特流可以通过网络传输（流媒体）或文件存储（本地播放）到达解码端。

**推理端全流程都是解码/生成操作，零编码**：

```
比特流（编好的文件）
    ↓  熵解码（解压缩，非编码）
  v̂（重建向量）
    ↓  哈希展开（确定性映射，非训练）
  完整 LoRA 参数 → 注入预训练模型
    ↓  纯噪声 → 扩散采样 N 步
  重建视频
```

### Q: 比特流是整个模型共享还是每个视频独立？

**每个视频有自己独立的比特流。** 模型和算法共享，视频内容不共享。

| 组件 | 共享？ | 说明 |
|------|--------|------|
| **冻结基础模型**（Wan-2.1 1.3B） | ✅ 全部视频共享 | 预训练好，人人都部署，不需要传 |
| **PRNG 哈希种子** | ✅ 全部视频共享 | 固定算法，解码端内置 |
| **熵模型参数 ψ** | ✅ 全部视频共享 | 8 个标量，提前训练好的全局参数 |
| **比特流（v̂ + caption）** | ❌ 每个视频独立 | **这才是真正传输的内容** |

类比：解码器软件（共享） vs. 视频文件（独立），和 `.mp4` 的逻辑完全一致。

### Q: 本文是视频压缩还是文生视频？prompt 的作用是什么？

**本文是视频压缩（编解码器），不是文生视频。** 输入是一段已有的视频，输出是压缩后的比特流。

Prompt/caption 的角色：
- **不是**用来「文生视频」的生成指令
- **是**编码时用 VLM 从原始视频自动提取的语义辅助信息
- 在扩散去噪时作为条件信号引导重建，帮助模型还原语义内容
- 它随比特流一起传给解码端（占不到 1% 的额外开销）

真实场景：

```
内容方（有原始视频，如电影公司母带）
  ├── 原始视频
  ├── VLM 自动打 caption："橘猫玩木地板上玩红色毛线球"
  ├── 编码：原始视频 + caption → 比特流
  └── 分发到 CDN

用户方（想观看这个视频）
  ├── 从 CDN 下载比特流
  ├── 解码：比特流 → 重建视频
  └── 播放
```

不能靠一句 prompt 凭空变出比特流，就像不能靠一句描述凭空变出一个 `.mp4` 文件。
