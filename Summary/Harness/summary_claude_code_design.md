# Dive into Claude Code: AI Agent 系统设计空间分析

## 论文概述

本文通过分析 Claude Code 的 TypeScript 源码（v2.1.88），系统地阐述了其完整架构设计，并与两个开源 AI Agent 系统（OpenClaw 和 Hermes Agent）进行对比分析，揭示了生产级 AI Agent 系统面临的核心设计问题及其解决方案。

---

## 五大设计价值观

Claude Code 的架构设计源于以下五大人类价值观：

1. **人类决策权威（Human Decision Authority）**：人类保留对系统行为的最终决策权，通过权限层级（Anthropic → 运营商 → 用户）正式化权威分配
2. **安全、安全和隐私（Safety, Security, and Privacy）**：保护人类及其代码、数据和基础设施免受伤害，即使人类疏忽或犯错时也是如此
3. **可靠执行（Reliable Execution）**：Agent 能准确理解人类意图，在时间上保持一致性，并支持在声明成功前验证工作
4. **能力增强（Capability Amplification）**：系统显著增加人类每单位努力和成本所能完成的任务量
5. **情境适应性（Contextual Adaptability）**：系统适应用户的特定情境（项目、工具、约定和技能水平），并随时间不断改进

---

## 十三项设计原则

这些价值观通过十三项设计原则来实现：

| 原则 | 服务的价值观 | 解决的设计问题 |
|------|-------------|---------------|
| 拒绝优先 + 人工升级 | 权威、安全 | 未识别的行为应允许、阻止还是升级给人类？ |
| 渐进信任谱 | 权威、适应性 | 固定权限级别还是用户随时间演进的信任谱？ |
| 深度防御 + 分层机制 | 安全、权威、可靠性 | 单一安全边界还是多重重叠的边界？ |
| 外部化可编程策略 | 安全、权威、适应性 | 硬编码策略还是带生命周期钩子的外部配置？ |
| 上下文作为稀缺资源 + 渐进管理 | 可靠性、能力 | 绑定资源约束是什么，如何管理？ |
| 追加式持久状态 | 可靠性、权威 | 可变状态、检查点快照还是追加式日志？ |
| 最小脚手架 + 最大操作架构 | 能力、可靠性 | 投资于脚手架端推理还是让模型自由推理？ |
| 价值观优先于规则 | 能力、权威 | 刚性决策程序还是确定性护栏下的情境判断？ |
| 可组合多机制扩展性 | 能力、适应性 | 统一扩展 API 还是分层机制？ |
| 可逆性加权风险评估 | 能力、安全 | 所有操作相同监督还是可逆操作更轻量？ |
| 透明文件式配置和内存 | 适应性、权威 | 不透明数据库、嵌入检索还是用户可见文件？ |
| 隔离子代理边界 | 可靠性、安全、能力 | 子代理共享父上下文还是隔离运行？ |
| 优雅恢复和韧性 | 可靠性、能力 | 错误时硬失败还是静默恢复？ |

## 架构概览

### 七大功能组件

1. **用户**：提交提示、批准权限、审查输出
2. **接口层**：交互式 CLI、无头 CLI、Agent SDK、IDE/桌面/浏览器集成，所有界面共享同一循环
3. **Agent 循环**：模型调用、工具调度、结果收集的迭代循环，实现为 `queryLoop` 异步生成器
4. **权限系统**：拒绝优先规则评估、auto-mode ML 分类器、钩子拦截
5. **工具池**：最多 54 个内置工具 + MCP 提供的工具
6. **状态与持久化**：追加式 JSONL 会话转录、全局提示历史、子代理侧链文件
7. **执行环境**：Shell 执行（可选沙箱）、文件系统操作、Web 获取、MCP 服务器连接

### 五层子系统架构

| 层级 | 职责 | 关键文件 |
|------|------|---------|
| **表面层** | 入口点和渲染 | `src/entrypoints/`, `src/screens/`, `src/components/` |
| **核心层** | Agent 循环、压缩管道 | `query.ts` |
| **安全/动作层** | 权限系统、钩子、扩展性、工具、沙箱、子代理 | `permissions.ts`, `tools.ts`, `hooks.ts` |
| **状态层** | 上下文组装、运行时状态、持久化、内存、侧链 | `context.ts`, `sessionStorage.ts`, `claudemd.ts` |
| **后端层** | 执行后端、外部资源 | `src/tools/`, `services/mcp/`, `src/remote/` |

## 关键子系统分析

### 1. Agent 循环（Turn Execution）

核心是一个简单的 while 循环，每轮执行以下步骤：
1. 设置解析 → 2. 可变状态初始化 → 3. 上下文组装 → 4. 模型前上下文塑形器 → 5. 模型调用 → 6. 工具使用调度 → 7. 权限门 → 8. 工具执行和结果收集 → 9. 停止条件

**五层上下文塑形器**（按顺序执行，由轻到重）：
- **预算缩减**（Budget reduction）：每条工具结果的大小限制
- **Snip**：轻量级历史修剪
- **Microcompact**：细粒度缓存感知压缩
- **Context collapse**：读时虚拟投影（不修改存储的历史）
- **Auto-compact**：完整模型生成摘要（最后手段）

### 2. 工具授权（Tool Authorization）

**七种权限模式**：
- `plan`：模型必须创建计划，经用户批准后执行
- `default`：标准交互模式，大多数操作需要用户批准
- `acceptEdits`：工作目录内的编辑和特定文件系统命令自动批准
- `auto`：ML 分类器评估请求（需要 `TRANSCRIPT_CLASSIFIER` 特性标志）
- `dontAsk`：抑制提示，原本会提示的操作自动拒绝
- `bypassPermissions`：跳过大多数权限提示，但安全关键检查仍适用
- `bubble`：子代理权限升级到父终端的内部模式

**七层安全防护**：
1. 工具预过滤 → 2. 拒绝优先规则评估 → 3. 权限模式约束 → 4. Auto-mode 分类器 → 5. Shell 沙箱 → 6. 会话恢复时不恢复权限 → 7. 钩子拦截

### 3. 扩展性（Extensibility）

四种扩展机制，按上下文成本递增排列：

| 机制 | 独特能力 | 上下文成本 | 插入点 |
|------|---------|-----------|-------|
| **MCP 服务器** | 外部服务集成（多传输） | 高（工具 schema） | `model()`: 工具池 |
| **插件** | 多组件打包 + 分发 | 中（可变） | 所有三个点 |
| **技能（Skills）** | 领域特定指令 + 元工具调用 | 低（仅描述） | `assemble()`: 上下文注入 |
| **钩子（Hooks）** | 生命周期拦截 + 事件驱动自动化 | 默认零 | `execute()`: 工具前后 |

**工具池组装五步流程**：
1. 基础工具枚举 → 2. 模式过滤 → 3. 拒绝规则预过滤 → 4. MCP 工具集成 → 5. 去重

### 4. 上下文构建与内存（Context and Memory）

**上下文窗口组装来源**：
- 系统提示、环境信息（git 状态）、CLAUDE.md 层次结构、路径范围规则、自动记忆、工具元数据、对话历史、工具结果、压缩摘要

**CLAUDE.md 四层记忆层次**：
1. **托管记忆**（`/etc/claude-code/CLAUDE.md`）：系统级策略
2. **用户记忆**（`~/.claude/CLAUDE.md`）：个人全局指令
3. **项目记忆**（`CLAUDE.md`, `.claude/CLAUDE.md`, `.claude/rules/*.md`）：代码库中的指令
4. **本地记忆**（`CLAUDE.local.md`）：gitignored，项目特定私有指令

**关键设计决策**：CLAUDE.md 内容作为用户消息（而非系统提示）传递，使合规性具有概率性而非确定性；权限规则提供确定性强制层。

### 5. 子代理委托（Subagent Delegation）

**六种内置子代理类型**：
- Explore（探索）、Plan（规划）、General-purpose（通用）、Claude Code Guide（指南）、Verification（验证）、Statusline-setup（状态栏配置）

**隔离模式**：
- **Worktree**：创建临时 git worktree，子代理有独立仓库副本
- **Remote**（内部）：在远程环境中启动
- **In-process**（默认）：共享文件系统但操作在隔离的对话上下文中

**侧链转录**：每个子代理写入独立的 `.jsonl` 文件，仅将最终摘要返回给父代理，防止子代理内容膨胀父上下文。

### 6. 会话持久化（Session Persistence）

**三种持久化通道**：
1. **会话转录**：包含用户、助手、附件、系统消息及压缩元数据，项目范围，每个会话一个文件
2. **全局提示历史**：仅用户提示，存储在 `history.jsonl`
3. **子代理侧链**：每个子代理独立的 `.jsonl` + `.meta.json` 文件

**关键安全设计**：会话恢复和分叉不恢复会话范围的权限，每次会话重新建立信任。

## 与 OpenClaw 和 Hermes Agent 的对比

| 维度 | Claude Code | OpenClaw | Hermes Agent |
|------|-------------|----------|--------------|
| **系统范围** | CLI/IDE 编码架构，每会话短暂进程 | 持久 WS 网关守护进程，多通道控制平面 | 单一 Python 进程，角色由入口点决定 |
| **信任模型** | 拒绝优先每动作规则评估 + 可选 ML 分类器，7 种权限模式 | 单一可信操作员，DM 配对和白名单，可选沙箱 | 每动作批准（3 种模式）+ 无条件安全底线 |
| **Agent 运行时** | 迭代异步生成器 `queryLoop` | 嵌入式 `pi-coding-agent` SDK 运行器 | 同步 while 循环，线程池并发 |
| **扩展架构** | 4 种机制（MCP、插件、技能、钩子） | Manifest-first 插件系统（12 种能力类型） | 5 种扩展面（含可插拔内存和模型后端） |
| **内存和上下文** | CLAUDE.md 4 层 + 5 层压缩管道 | 工作空间引导文件 + 记忆系统（梦想、每日笔记） | 辅助 LLM 摘要器 + 持久 MEMORY.md |
| **多代理和路由** | 任务委托子代理 + worktree 隔离 | 多代理路由（独立代理）+ 子代理委托 | `delegate_task` 工具 + Kanban 子系统 |

## 设计哲学与权衡

### 核心设计哲学

Claude Code 的架构投资于**确定性基础设施**而非**决策脚手架**。代码库中估计只有约 1.6% 是 AI 决策逻辑，剩余 98.4% 是操作基础设施。这反映了一个信念：能力越来越强的模型更受益于丰富的操作环境，而非限制其选择的框架。

### 价值张力

| 价值对 | 张力 | 证据 |
|--------|------|------|
| 权威 × 安全 | 批准疲劳 vs. 保护 | 93% 批准率削弱人类警惕性 |
| 安全 × 能力 | 性能 vs. 防御深度 | >50 子命令时跳过逐条拒绝检查 |
| 适应性 × 安全 | 扩展性 vs. 攻击面 | 多个 CVE 利用钩子和 MCP 服务器的预信任初始化 |
| 能力 × 适应性 | 主动性 vs. 干扰 | 任务增加 12-18% 但高频时偏好下降 |
| 能力 × 可靠性 | 速度 vs. 一致性 | 有界上下文限制全代码库感知 |

### 架构权衡

1. **安全 vs. 自治**：权限模式构成单调递减的安全梯度，用户习惯化而非主动选择模式
2. **上下文效率 vs. 透明度**：五层压缩有效但对用户不可见，压缩决策受提示缓存影响
3. **简单性 vs. 扩展性**：四种扩展机制支持丰富定制但产生组合交互，难以预测

---

## 六大开放设计方向

1. **沉默故障与可观测性-评估差距**：78% 的 AI 故障是不可见的，需要 generator-evaluator 分离等脚手架
2. **跨会话持久化**：在静态指令和单会话转录之间需要持久状态层，支持纵向人机协作关系
3. **架构边界演进**：探索代理在哪里运行、何时行动、作用于什么、与谁协调
4. **时间范围扩展**：从单会话到多会话程序，需要协调原语超越会话、子代理和内存
5. **治理与监督**：AI 监管要求架构暴露日志、透明度和人工监督接口
6. **长期开发者能力**：短期能力增强是否损害长期理解和代码库一致性

## 结论

Claude Code 在 AI Agent 设计空间中占据一个清晰的设计点：在密集的确定性架构（权限、工具路由、上下文压缩、扩展性、会话恢复）中给予模型广泛的局部自治。与 OpenClaw 和 Hermes Agent 的对比表明，相同的设计问题在不同部署上下文中产生不同答案。最关键的开放问题不是如何增加更多自治，而是**如何随时间保持人类能力**。

---

## 关键见解

1. **架构即答案**：生产级编码代理可被理解为对一系列反复出现的设计问题的回答
2. **基础设施优先**：当模型能力增强时，周围操作架构的质量成为主要差异化因素
3. **设计空间是分层的**：网关级系统和任务级架构可以在多个位置组合，而非互斥替代
4. **长期能力是第一类设计问题**：未来系统应将可持续性差距视为核心设计问题，而非下游评估指标
