# DeepSeek-V4: Towards Highly Efficient Million-Token Context Intelligence

- **论文链接**: [arXiv:2606.19348](https://arxiv.org/abs/2606.19348)
- **作者**: DeepSeek-AI
- **发布时间**: 2026年6月
- **关键词**: 混合稀疏注意力、百万级上下文、MoE、Muon优化器、流形约束超连接、FP4量化

---

## 1. 概述

DeepSeek-V4 是 DeepSeek 系列的下一代大语言模型，核心目标是**突破超长上下文的效率瓶颈**，原生支持 **100万 token 上下文长度**。发布两个预览版本：

| 模型 | 总参数 | 激活参数 | 预训练 Token 数 |
|------|--------|----------|:----:|
| **DeepSeek-V4-Pro** | 1.6T | 49B | 33T |
| **DeepSeek-V4-Flash** | 284B | 13B | 32T |

每种模型支持三种推理模式：**Non-think**、**Think High**、**Think Max**。

在 100万 token 上下文中，DeepSeek-V4-Pro 仅需 V3.2 的 **27% 推理 FLOPs** 和 **10% KV Cache**；DeepSeek-V4-Flash 则仅为 **10% FLOPs** 和 **7% KV Cache**。

## 2. 架构创新

### 2.1 混合注意力：CSA + HCA

V4 保留了 DeepSeekMoE 和 Multi-Token Prediction (MTP)，同时引入交错混合的两种高效注意力机制：

#### CSA（Compressed Sparse Attention，压缩稀疏注意力）

- **压缩阶段**：将每 $m=4$ 个 token 的 KV entries 压缩为 1 个压缩 entry（通过带可学位置偏置的 softmax 加权求和）
- **稀疏选择**：Lightning Indexer 计算压缩 KV 块与 query 的索引分数 $I_{t,s}$，通过 Top-k 选择 $k=1024$（Pro）/ $512$（Flash）个块
- **MQA + 分组输出投影**：压缩 KV 在多个 query head 间共享，输出分组投影降低计算开销
- 附带滑动窗口注意力分支（$n_{\text{win}}=128$）增强局部依赖建模

#### HCA（Heavily Compressed Attention，重度压缩注意力）

- 压缩比率 $m'=128$（远超 CSA 的 $m=4$），不采用稀疏选择
- 同样使用 MQA + 分组输出投影 + 滑动窗口
- 负责大幅降低整体 KV Cache 和计算量

**混合配置**：Pro 共 61 层（前 2 层用 HCA，后续 CSA/HCA 交错），Flash 共 43 层（前 2 层用纯滑动窗口注意力，后续 CSA/HCA 交错）。

**其他关键技术**：
- 部分 RoPE（最后 64 维）+ RoPE 反向应用到注意力输出以保证相对位置编码
- Query/KV entry 预 RMSNorm 防止注意力 logit 爆炸
- **Attention Sink**：可学习的 sink logit 使注意力头可调节总注意力分数不为 1
- 混合精度存储：RoPE 维度用 BF16，其余用 FP8，KV Cache 减半

#### 2.1.1 与 DeepSeek-V3.2 DSA 的对比

V4 的注意力机制是对 V3.2 的 DSA (DeepSeek Sparse Attention) 的重大改进。核心变化是**引入压缩步骤**，将 DSA 应用于压缩后的 KV entries，而非原始 token。

| 维度 | V3.2 (DSA) | V4 (CSA + HCA) |
|------|------------|----------------|
| **注意力类型** | 单一 DSA | 混合注意力（CSA + HCA 交错） |
| **KV 处理** | 直接对 token 进行稀疏选择 | **先压缩 KV**，再进行稀疏/稠密注意力 |
| **压缩比率** | 无压缩 | CSA: $m=4$, HCA: $m'=128$ |
| **选择粒度** | Token 级（2048 tokens） | 压缩块级（CSA: 1024/512 压缩块） |
| **Lightning Indexer** | 在原始 token 上计算索引分数 | 在**压缩 KV** 上计算索引分数 |
| **MQA 模式** | MLA + MQA（隐式） | 显式 MQA（压缩 KV 共享）+ 分组输出投影 |
| **局部建模** | 未明确强调 | **滑动窗口注意力**（$n_{\text{win}}=128$） |
| **Attention Sink** | 无 | 有（可学习 sink logit） |
| **RoPE** | 完整 RoPE | **部分 RoPE**（仅最后 64 维） |
| **Query 归一化** | 无 | 有（RMSNorm on queries and KV entries） |
| **输出投影** | 标准投影 | **分组输出投影**（降低计算开销） |

**详细对比**：

1. **V3.2 的 DSA**：
   - Lightning Indexer 计算 token 级索引分数：$I_{t,s} = \sum_{j=1}^{H^I} w_{t,j}^I \cdot \text{ReLU}(\mathbf{q}^{I}_{t,j} \cdot \mathbf{k}^{I}_s)$
   - 选择 top-k tokens（2048 个）
   - 基于 MLA 的 MQA 模式

2. **V4 的 CSA（基于 DSA 改进）**：
   - **步骤 1 - KV 压缩**：将每 $m=4$ 个 token 的 KV entries 压缩为 1 个压缩 entry
     $$C^{\text{Comp}}_i = \sum_{j=mi}^{m(i+1)-1} S^a_j \odot C^{a}_{j} + \sum_{j=m(i-1)}^{mi-1} S^b_j \odot C^{b}_{j}$$
   - **步骤 2 - 稀疏选择**：在压缩 KV 上应用 Lightning Indexer（改进版）
     - Indexer queries 通过低秩投影生成：$\mathbf{c}_{t}^{Q} = \mathbf{h}_{t} \cdot W^{DQ}$, then $\mathbf{q}_{t}^{I} = \mathbf{c}_{t}^{Q} \cdot W^{IUQ}$
     - 压缩索引器 keys：$K^{\text{IComp}}$ 同样通过压缩得到
   - **步骤 3 - MQA**：压缩 KV 在多个 query head 间共享
   - **步骤 4 - 分组输出投影**：降低输出投影的计算开销

3. **V4 的 HCA（重度压缩，无稀疏选择）**：
   - 压缩比率 $m'=128$（远超 CSA 的 $m=4$）
   - **不使用稀疏选择**，直接对压缩 KV 做**稠密注意力**
   - 同样使用 MQA + 分组输出投影
   - 大幅降低整体 KV Cache 和计算量

**效率提升**：

| 指标 | V3.2 | V4-Pro | V4-Flash |
|------|------|--------|----------|
| 单 Token 推理 FLOPs (1M 上下文) | 100% | **27%** | **10%** |
| KV Cache 大小 (1M 上下文) | 100% | **10%** | **7%** |

**设计动机**：

1. **压缩的必要性**：
   - V3.2 的 DSA 仍需要 $O(L^2)$ 的索引器计算（虽然主注意力降至 $O(Lk)$）
   - V4 通过压缩将序列长度降为 $\frac{1}{m}$（CSA）或 $\frac{1}{m'}$（HCA），进一步降低索引器和注意力的计算量

2. **混合注意力的优势**：
   - CSA：平衡稀疏注意力的效率和对重要 token 的检索能力
   - HCA：通过重度压缩大幅降低 KV Cache，适合不需要精细检索的场景
   - 交错配置：两种注意力互补，兼顾效率和性能

3. **工程优化**：
   - 分组输出投影：降低 $cn_h \rightarrow d$ 的投影开销
   - 部分 RoPE：减少位置编码的计算和存储开销
   - 混合精度：FP8 存储 + BF16 RoPE，KV Cache 减半

#### 2.1.2 与 MiniMax MSA 的对比

CSA 的压缩步骤与 MSA 的块级选择有相似之处，但存在本质区别：

| 维度 | MSA（块级选择） | CSA（压缩 + 选择） |
|------|-----------------|-------------------|
| **选择前的处理** | 无（直接使用原始 KV） | **压缩**（m=4 个 token → 1 个 entry） |
| **信息保留（选中时）** | **完整保留**（使用原始 KV） | **有损压缩**（使用压缩 entry） |
| **选择粒度** | 块（128 tokens） | 压缩块（1 个 entry 代表 4 tokens） |
| **选择后的计算** | 在原始 token 上计算注意力 | 在压缩 entry 上计算注意力 |
| **类比** | "整块提取或整块放弃" | **"先榨汁再选择是否喝"** |

**核心区别**：

1. **"整块放弃"**：✅ 相似
   - MSA：未选中的块 → 整块放弃
   - CSA：未选中的压缩 entry → 对应的 m 个 token 整块放弃

2. **"整块提取"**：❌ 不相似
   - MSA：选中的块 → **提取完整的原始 token**（无信息损失）
   - CSA：选中的压缩 entry → 提取的是**压缩后的表示**（有信息损失）

**本质区别**：
- MSA：**选择粒度粗**，但**保留原始信息**（硬选择，All-or-nothing）
- CSA：**选择粒度细**（压缩后），但**信息是压缩的**（软聚合，加权求和）

**设计权衡**：

| 设计选择 | 效率 | 性能 |
|---------|------|------|
| **MSA（无压缩）** | 较低（块大小 128，仍需处理 128 个 token） | 较高（保留完整信息） |
| **CSA（有压缩）** | **极高**（压缩比 4:1，序列长度降为 1/4） | 较低（压缩导致信息损失） |

CSA 通过**可学习压缩权重**（$S^a_j, S^b_j$）和**Lightning Indexer on compressed KV** 来弥补压缩的信息损失。

#### 2.1.3 与 MLA 的兼容性问题

**简短回答**：理论上可以兼容（正交设计），但 DeepSeek-V4 实践中未采用 MLA，而是使用了 MQA。

**详细分析**：

1. **MLA 和 CSA 的压缩方式是正交的**：

| 维度 | MLA (Multi-head Latent Attention) | CSA (Compressed Sparse Attention) |
|------|----------------------------------|----------------------------------|
| **压缩维度** | **特征维度**（低秩分解） | **序列维度**（时间压缩） |
| **压缩对象** | KV cache（存储 `c_t^KV` 而非完整 KV） | 多个 token 的 KV（m 个 → 1 个） |
| **压缩目标** | 降低 KV cache 大小 | 降低序列长度 + 索引器负担 |
| **是否正交** | ✅ 是 | ✅ 是 |

2. **DeepSeek-V4 实际使用的是 MQA，而非 MLA**：

从论文第 429-430 行和第 481-482 行可以明确看到：
> "performs core attention in a Multi-Query Attention (MQA) manner"
> "HCA also employs the shared KV MQA"

**V4 的设计选择**：
- ❌ 未使用 MLA
- ✅ 使用 MQA（多查询注意力，所有 query head 共享 KV）
- ✅ 使用查询压缩（`c_t^Q`），类似 MLA 的哲学但应用于查询

3. **为什么 V4 未采用 MLA？**

| 可能原因 | 说明 |
|---------|------|
| **CSA 的压缩已足够** | 时间压缩（m=4 或 m'=128）已大幅降低 KV Cache，MLA 的额外特征压缩边际收益递减 |
| **简化实现** | MQA 比 MLA 更简单（MLA 需要特殊的 RoPE 处理） |
| **设计哲学转变** | V2/V3 用特征压缩（MLA），V4 转向序列压缩（CSA/HCA） |

4. **CSA 中的 MLA 元素**：

虽然 V4 未使用 MLA，但 CSA 的查询生成使用了**类似 MLA 的低秩压缩哲学**：

```latex
% 第 413-416 行：查询的低秩压缩
c_t^Q = h_t * W^DQ          % 降维到 d_c
q_t^I = c_t^Q * W^IUQ      % 升维到索引器查询
q_t = c_t^Q * W^UQ          % 升维到注意力查询
```

**对比 MLA**：
- MLA：KV 通过低秩投影生成（`c_t^KV`）
- CSA：查询通过低秩投影生成（`c_t^Q`）
- **相似哲学**：低秩压缩减少计算和存储

5. **如果要将 CSA 与 MLA 结合，如何设计？**

**理论方案**：
```
输入 → MLA (KV 低秩压缩) → CSA (时间压缩) → 稀疏选择 → MQA
```

**步骤**：
1. **MLA 步骤**：对每个 token，计算低秩 KV `c_t^KV`
2. **CSA 压缩步骤**：将每 m 个 token 的 `c_t^KV` 压缩为 1 个 entry
3. **稀疏选择**：在压缩后的 KV 上应用 Lightning Indexer
4. **MQA**：共享 KV 进行注意力计算

**挑战**：
- MLA 的 RoPE 处理需要与 CSA 的时间压缩协调
- 压缩 `c_t^KV` 时，如何保留位置信息？
- 实现复杂度较高

**总结**：

| 问题 | 答案 |
|------|------|
| **CSA 与 MLA 理论上兼容吗？** | ✅ 是（正交压缩，一个在特征维度，一个在序列维度） |
| **DeepSeek-V4 使用了 MLA 吗？** | ❌ 否（使用 MQA 替代） |
| **为什么未采用 MLA？** | 1) CSA 压缩已足够；2) 简化实现；3) 设计哲学转变 |
| **CSA 中有 MLA 的元素吗？** | ✅ 有（查询的低秩压缩 `c_t^Q`） |
| **未来可能结合吗？** | 可能，但需要解决工程挑战 |

### 2.2 流形约束超连接 (mHC)

- 将残差流宽度扩展 $n_{\text{hc}}=4$ 倍
- **核心创新**：约束残差变换矩阵 $B_l$ 到双随机矩阵流形 $\mathcal{M}$（Birkhoff 多面体），保证 $\|B_l\|_2 \leq 1$，非膨胀变换增强训练稳定性
- 通过 Sinkhorn-Knopp 算法（$t_{\text{max}}=20$ 次迭代）将 $B_l$ 投影到双随机矩阵
- 输入/输出映射通过 Sigmoid 约束为非负有界
- 动态参数化：输入依赖 + 静态偏置双组分生成
- 训练墙钟时间开销仅 6.7%

### 2.3 Muon 优化器

- 对大部分模块使用 Muon 替代 AdamW（embedding/head/Norm 仍用 AdamW）
- **混合 Newton-Schulz 迭代**：前 8 步用 $(3.4445, -4.7750, 2.0315)$ 快速收敛，后 2 步用 $(2, -1.5, 0.5)$ 精确稳定奇异值
- Nesterov 技巧 + RMS 重缩放
- 使用 RMSNorm 替代 QK-Clip 防止注意力 logit 爆炸

## 3. 基础设施优化

### 3.1 细粒度通信-计算重叠 EP

- 提出 **MegaMoE** 核：将 MoE 的 Dispatch/Linear-1/Linear-2/Combine 融合为单流水线
- 基于 Wave 的专家调度：通信与计算完全重叠
- 加速比：通用推理 $1.50\sim1.73\times$，RL rollout 场景最高 $1.96\times$
- 关键洞察：当 $C/B \leq 2d = 6144$ FLOPs/Byte 时，通信可完全隐藏

### 3.2 TileLang 高效核开发

- 使用 TileLang DSL 开发融合核，平衡开发效率和运行时性能
- **Host Codegen**：将主机端逻辑编译为生成代码，调用开销从百微秒降至 <1μs
- **SMT 求解器辅助整数分析**：集成 Z3 求解器（QF_NIA）提升向量化、边界分析等优化能力

### 3.3 批量不变与确定性核库

- **批量不变性**：双核策略，第一个核在单 SM 内完成解码，第二个核利用分布式共享内存处理尾部 wave
- **确定性**：注意力反向使用独立累积 buffer、MoE 反向使用 token 顺序预处理 + buffer 隔离、mHC 矩阵乘法使用 split-k + 确定性归约
- 端到端位级可复现，便于调试和稳定性分析

### 3.4 训练框架增强

- Muon 混合 ZeRO：背包算法分配稠密参数，MoE 参数扁平化分布
- MoE 梯度 BF16 随机取整 + all-to-all 通信替代 reduce-scatter
- mHC 选择性重计算 + DualPipe 调整
- **两阶段上下文并行**（CP）：处理压缩注意力的不规则长度
- **张量级激活检查点**：基于 TorchFX 的细粒度控制

### 3.5 推理框架：异构 KV Cache 管理

- **状态缓存**：SWA + 未完成压缩的尾 token 固定池
- **经典缓存**：每块覆盖 $\text{lcm}(m, m')$ 个原始 token
- **磁盘 KV Cache 存储**：前缀复用，三种 SWA 策略（Full/Periodic/Zero Caching）
- **Quick Instruction**：通过特殊 token 并行执行搜索触发、意图识别等辅助任务，消除重复预填充

### 3.6 DSec：弹性计算沙箱平台

- 生产级智能体沙箱，支持数十万并发实例
- 四种执行基底统一接口：Function Call / Container / microVM (Firecracker) / fullVM (QEMU)
- 基于 EROFS 的分层镜像快速加载
- 轨迹日志支持客户端快进 + 故障恢复

## 4. 预训练

- **数据**：32T+ token，强调长文档（科学论文、技术报告）、多语言长尾知识、反模板化过滤
- **训练**：序列长度从 4K → 16K → 64K → 1M 渐进扩展
- **稀疏注意力引入**：前 1T token 用稠密注意力，在 64K 阶段引入，先预热 Lightning Indexer，再全量稀疏训练
- **训练稳定性**：
  - **Anticipatory Routing（前瞻路由）**：用历史参数 $t-\Delta t$ 计算路由，解耦骨干网络与路由网络更新，损失尖峰时自动激活
  - **SwiGLU Clamping**：线性分量钳制 $[-10,10]$，门控分量上限 10
- V4-Flash-Base 已在多数基准上超越 V3.2-Base（参数更少），V4-Pro-Base 全面领先

## 5. 后训练

### 5.1 训练范式

- **专家训练**：多个领域（数学、编程、Agent、指令跟随等）独立训练专家模型
  - DeepSeek-V3 方法：初始 SFT + GRPO RL
  - 每个领域支持 thinking/non-thinking 模式
- **多教师 On-Policy Distillation (OPD)**：统一学生模型学习多个专家教师的输出分布
  - 全词汇表 logit 级别 KL 散度（非简化 token 级估计），梯度更稳定
  - 超过 10 个教师模型

### 5.2 FP4 量化感知训练 (QAT)

- MoE 专家权重 FP4 量化：FP4→FP8 反量化无损（FP8 额外 2 个指数位覆盖细粒度缩放）
- CSA 索引器 QK 路径 FP4 计算
- 索引分数 FP32→BF16：Top-k 选择器 2× 加速，99.7% KV 召回率

### 5.3 交错思考 (Interleaved Thinking)

- **工具调用场景**：跨用户消息回合**保留所有推理内容**（利用 1M 上下文优势）
- **通用对话场景**：新用户消息到达时丢弃历史推理内容

### 5.4 推理模式

| 模式 | 特点 | 适用场景 |
|------|------|----------|
| Non-think | 快速直觉响应 | 日常任务 |
| Think High | 有意识逻辑分析 | 复杂问题求解 |
| Think Max | 最大化推理深度（注入特殊系统提示） | 探索推理边界 |

### 5.5 工具调用格式

采用 XML 格式的 `|DSML|` 特殊 token 工具调用模式，减少转义失败和工具调用错误。

### 5.6 生成式奖励模型 (GRM)

- 替代标量奖励模型：Actor 网络本身充当 GRM，联合优化评估和生成能力
- 仅需少量人工标注即可泛化

## 9. Q&A：训练和推理阶段的量化方式

### Q: DeepSeek-V4 在训练和推理阶段分别采用了怎样的量化方式？

**简短回答**：预训练阶段基本沿用 V3 的 FP8 混合精度训练框架；后训练阶段引入 **FP4 量化感知训练（QAT）** 对 MoE 专家权重和 CSA 索引器 QK 路径进行量化；推理阶段采用多层次混合精度策略——KV Cache 用 BF16+FP8 混合存储，Lightning Indexer 计算用 FP4，路由专家权重用 FP4，索引分数用 BF16。

---

### 一、训练阶段

#### 1.1 预训练阶段

- 预训练**未使用 FP4 量化**，主要沿用 V3 的混合精度训练框架
- 骨干网络使用 **FP8 混合精度训练**（FP8 forward/backward，FP32 主权重）
- 优化器使用 **Muon**（大部分模块）+ **AdamW**（embedding/head/Norm 层），均维护 FP32 主权重
- 预训练的序列长度从 4K → 16K → 64K → 1M 渐进扩展

#### 1.2 后训练阶段：FP4 量化感知训练（QAT）

在后训练阶段（包含 RL 和 OPD），引入 FP4 QAT，量化对象包括：

| 量化对象 | 量化精度 | 目的 |
|----------|:--------:|------|
| **MoE 路由专家权重** | FP4 (MXFP4, E2M1) | 减少 GPU 显存占用，加速推理 |
| **CSA 索引器 QK 路径** | FP4 | 加速长上下文下注意力分数计算 |
| **索引分数 $I_{:, :}$** | FP32 → BF16 | Top-k 选择器 2× 加速 |

**QAT 训练流程详解**：

1. **前向传播**：
   - FP32 主权重 → **量化到 FP4** (MXFP4, tile 大小 $1\times 32$)
   - FP4 权重 → **反量化到 FP8** (E4M3) 进行计算
   - 注意：**FP4→FP8 反量化是无损的**，因为 FP8 比 FP4 多 2 个指数位，细粒度缩放信息可被 FP8 的动态范围完全吸收
   - 条件：FP4 子块缩放因子之比不超过 FP8 量化块（$128\times 128$）的阈值（论文实证当前权重满足此条件）

2. **反向传播**：
   - 梯度基于前向的 FP8 权重计算
   - 通过 **直通估计器（STE）** 直接将梯度传回 FP32 主权重
   - 无需对转置权重重新量化，梯度更新等价于 $\theta_{\text{FP32}} \leftarrow \theta_{\text{FP32}} - \eta \cdot \nabla \mathcal{L}(\theta_{\text{FP8}})$

3. **RL 推理/Rollout 阶段**：
   - 直接使用**原生 FP4 量化权重**（非模拟量化）
   - 行为与在线部署完全一致
   - 实际降低 kernel 显存加载开销和显存占用

4. **QAT 覆盖范围**：
   - 学生模型 + 所有教师模型 + 参考模型均进行 QAT
   - 使得模型在训练中适应量化引入的精度损失

---

### 二、推理阶段

推理阶段采用**多层次混合精度策略**，在不同模块使用不同精度：

| 组件 | 精度 | 说明 |
|------|:----:|------|
| **KV Cache（RoPE 维度）** | BF16 | 最后 64 维使用 BF16 以保留位置精度 |
| **KV Cache（其余维度）** | FP8 | 非 RoPE 维度用 FP8 存储，KV Cache 相比纯 BF16 **减半** |
| **Lightning Indexer 注意力计算** | **FP4** | 索引器 QK 激活的缓存、加载、矩阵乘法全程 FP4 |
| **MoE 路由专家权重** | **FP4** (E2M1) | 推理时直接使用 FP4 量化权重 |
| **索引分数** | BF16 | 从 FP32 量化到 BF16，Top-k 选择器 2× 加速 |
| **骨干网络计算** | FP8 | 主要计算（非量化部分）延续 FP8 |

**精度选择的层次逻辑**：

```
精度层级（从高到低）：
  FP32 ─── 优化器主权重、embedding、head、Norm
  BF16 ─── KV Cache RoPE 维度、索引分数
  FP8  ─── 骨干计算、KV Cache 非 RoPE 维度
  FP4  ─── MoE 专家权重、CSA 索引器 QK 路径
```

**效率收益**（1M 上下文下与 V3.2 对比）：

| 指标 | V4-Pro | V4-Flash |
|------|:------:|:--------:|
| 单 Token 推理 FLOPs | 27% | 10% |
| KV Cache 大小 | 10% | 7% |

**FP4 × FP8 的理论潜力**：虽然在现有硬件上 FP4×FP8 和 FP8×FP8 的峰值 FLOPs 相同，但理论上在未来硬件上可实现额外 **1/3 的效率提升**。

### 三、量化方案总结

| 阶段 | 量化对象 | 方案 | 关键特性 |
|------|----------|------|----------|
| 预训练 | 骨干网络 | FP8 混合精度 | 沿用 V3 框架，FP32 主权重 + FP8 计算 |
| 后训练 QAT | MoE 专家权重 | FP4 QAT (STE) | FP4→FP8 无损反量化，复用 FP8 训练框架 |
| 后训练 QAT | CSA 索引器 QK | FP4 QAT | 索引器内全程 FP4，加速长上下文注意力 |
| 后训练 QAT | 索引分数 | FP32→BF16 | 99.7% KV 召回率，Top-k 2× 加速 |
| 推理 | KV Cache 存储 | BF16 + FP8 混合 | RoPE 维度 BF16，其余 FP8，减半显存 |
| 推理 | 索引器注意力 | FP4 | QK 激活全程 FP4 计算 |
| 推理 | MoE 路由专家 | FP4 权重 | 直接使用 FP4 量化权重 |
